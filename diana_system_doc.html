Analista de Sistemas

Varredura funcional — como o sistema se comporta

1

Pipeline de dados — sync diário às 17h30

O sistema executa três fetchers em sequência: inbound_fetcher.py (lê inbound_nutria via MySQL, user thomas), outbound_fetcher.py (lê nutricao_outbound, top 30 por Score DESC) e audit_fetcher.py (lê leads_controladoria_compliance_supply para modo Audit). Todos gravam no SQLite local diana_memory.db. Cron: 30 17 * * 1-5.

# Tabelas MySQL → SQLite inbound_nutria → inbound_leads (570 leads, score_geral, opened, clicked) nutricao_outbound → outbound_leads (top 30, Score DESC, respondeu, cargo_decisor) leads_ctrl_compliance→ audit_leads (campanha Audit — auditoria/tax/compliance/planej.financeiro/controladoria/supply, ~68 variantes EN↔PT)
2

Lógica de recomendação — dois modos

Modo Normal: retorna 5 inbound (score_geral DESC, source='both' primeiro) + 2 outbound (Score DESC). Modo Audit: ativado por palavras-chave (audit, auditoria, tax, compliance, controladoria, risk, supply, planejamento financeiro) — retorna 3 inbound filtrados por cargo + 1 outbound da tabela audit_leads. Match semântico PT↔EN ativado (ex: "Tax Manager" = "Gerente de Tax"). Filtros obrigatórios em ambos: reserva ativa de outro SDR (<7 dias), deal aberto no Pipedrive e Pre-Sales em estágio early.

3

Score inbound — composição dos 5 critérios

Score máximo: 97 pts. Campos: opened (0/15) — abriu email; clicked (0/25) — clicou em link, MAIOR PESO; empresa_potencial_usuario (8/12/15) — outros da empresa engajados; contagem_listas (8/12/15) — múltiplas listas; pipedrive — campo NÃO usado para filtro (incorreto no upstream), verificação feita em tempo real via pipedrive_monitor.py.

4

Reserva de leads por SDR — 7 dias

Ao recomendar, Diana grava em recomendacoes (email, sdr, data_recomendacao). Lead recomendado fica reservado para aquele SDR por 7 dias. Se outro SDR pedir o mesmo lead: "⚠️ Reservado para [SDR] até [data]". Expiração automática por comparação de datas na query SQL.

5

Regra Pipedrive — Won/Lost/Aberto/Pre-Sales

Diana verifica status via pipedrive_monitor.py antes de recomendar. ⚪ Sem histórico = pode prospectar. 🔴 Lost = pode reabordar. 🟡 Deal aberto nos funis Simon.AI/Training/Hub/Consulting/Pre-Sales = NÃO recomendar este lead. 🟢 Won = cliente ativo, segue para CS, não prospectar. ⚠️ Empresa em prospecção com outra pessoa da empresa = sinalizar contexto (não bloquear).

6

Memória de abordagens e cruzamento BOTH

SDRs registram resultado em linguagem livre — Diana interpreta e salva em abordagens (email, sdr, relato, status, canal_sugerido). Leads com email em ambas as bases (inbound + outbound) recebem source='both' — prioridade máxima nas recomendações, marcados com [BOTH ★].

7

Comunicação com o SDR — filtro interno silencioso

Pipedrive é filtro interno, não fonte de leads. Diana diz "Vou buscar no Inbound e Outbound..." — nunca "Vou checar Pipedrive em tempo real...". Status Pipedrive aparece nos cards via badges (⚪⚡🟡🟢⚠️), nunca anunciado antes das recomendações.

Arquiteto de Sistemas

Infraestrutura, isolamento e decisões técnicas

1

Arquitetura geral — isolamento por agente

Diana roda em isolamento total do Marcel (agente comercial). Cada agente tem: Linux user próprio (diana-hupdata), porta de gateway exclusiva (18790), porta msteams exclusiva (3979), env file separado (/opt/openclaw-diana.env) e banco SQLite independente. Princípio: falha em um agente não afeta o outro.

2

Stack tecnológica e modelo LLM

Gateway: OpenClaw (Node.js) gerencia sessões Teams e injeta SOUL.md como system prompt. Modelo: anthropic/claude-haiku-4-5 — forçado via openclaw.json, Opus bloqueado por custo (~15x mais caro). Comunicação Teams: Bot Framework via Azure App ID a0be4e53, tenant HupData. Python 3.12 com venv isolado em /home/diana-hupdata/.openclaw/venv/. Dependências: pymysql, cryptography.

# openclaw.json — modelo fixo "agents": { "defaults": { "model": "minimax/MiniMax-M3", "heartbeat": { "every": "0" } // desabilitado — reduz custo } } # dmPolicy: allowlist — só usuários autorizados
3

Estratégia de dados — MySQL → SQLite

Decisão arquitetural: não consultar MySQL em tempo real a cada mensagem. O pipeline sync (17h30) materializa os dados no SQLite local — consultas da Diana são sub-milissegundo. Exceção intencional: Pipedrive é consultado em tempo real via pipedrive_monitor.py pois status de deal muda ao longo do dia. Cache local + real-time pontual é o padrão estabelecido.

4

Gestão de custo — Haiku + SOUL.md

Principal vetor de custo: tamanho do SOUL.md (~350 linhas) carregado em todo contexto. Roadmap: enxugar para ~60 linhas core + carregar base de conhecimento (hupdata_cases.txt, process.yaml) sob demanda. Estimativa: redução de 50-60% no custo por sessão sem perda de qualidade. Interceptação de /model no SOUL.md evita upgrade acidental para Opus.

5

Sessões e isolamento de contexto

Cada SDR tem sessão DM privada e independente (dmScope: per-channel-peer, dmPolicy: allowlist). Bug histórico de cross-contamination de contexto entre sessões simultâneas foi resolvido migrando de dmPolicy: open para allowlist. Limpeza automática de trajectories >5MB via cron às 3h. TTL de sessão: 60 min.

6

Integração Azure / Teams

App Registration Azure (a0be4e53-7cab-449b-ad80-49c0b9b6a1e6), Single Tenant, secret via env file. Caddy faz strip do prefixo /diana antes de rotear para porta 3979 — padrão necessário pois OpenClaw msteams só aceita /api/messages sem prefixo. Allowlist de 8 Object IDs do Azure AD controla quem pode usar a Diana.
